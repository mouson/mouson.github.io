系統架構

  • Design Patterns
    Geekbang/ArchSummit: ArchSummit全球架构师峰会

  • Nisra16 你的 https 真的安全嗎？ - 一份把 https 安全性及基本原理方法以及用了不好的憑證會發生什麼問題，都解釋的很清楚的簡報，文中也提供了一些檢測工具可以參考使用。

    • 透過「非對稱式加密」協商出「對稱式加密的金鑰」，然後利用「對稱式加密」來將流量加密。
    • 好處：完整性、安全性、實用性、必要性、強制性
    • Let’s Encrypt 使用說明 (非官方)
    • 測試 HTTPS 安全：
      • Qualys SSL Labs 測試至少不要拿 F
      • /bin/bash based SSL/TLS tester: testssl.sh - 當需要測試的環境在區網時，可以用 testssl.sh 做測試
    • https 的缺點：
      • 建立連線的步驟較多，可能拖慢開啟速度
      • 大部分需要付費且不便宜
      • 簽發憑證的信任單位不被信任，則它所簽發的也會不被信任
      • 相容性問題，版本太舊的瀏覽器無法支援
      • http 一個 TCP 握手(一個 RTT)
      • https TCP 握手 + SSL 握手 (三個 RTT)
    • https 建議調整之重點：
      • 關閉支援 SSLv2, SSLv3
      • 移除不安全的 Cipher Strength (MD5, 低於 1024bit)
      • 開啟 HSTS (HTTP Strict Transport Security)
      • 開啟 Perfect Forward Secrecy
      • 如果 Weak Deffie-Hellman (DH) 問題，調整預設質數長度
      • 建議的調整工具：Cipherli.st - Strong Ciphers for Apache, nginx and Lighttpd
    • https 加速方案：
      • TLS False Start - Client 端發送 Change Cipher Spce Finished 時，一起接 application data 回來
      • Certificate-Chain - 考慮使用 ECC 憑證，壓低憑證大小
      • Session Resumption - 把交握後的對稱式金鑰存起來，減少一次 RTT
      • OCSP Stapling
